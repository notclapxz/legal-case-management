## 🔧 Cambios Implementados

### 1. Nuevo Formato de Códigos de Caso

**Problema Original**:
- Formato antiguo: `APELLIDO-##` (ej: AGUIRRE-01, AGUIRRE-02)
- No se podía distinguir entre diferentes personas con mismo apellido
- Ejemplo: Carlos Aguirre y Pedro Aguirre tendrían códigos idénticos

**Solución Implementada**:
- Formato nuevo para personas: `I.APELLIDO-##` (ej: C.AGUIRRE-01, P.AGUIRRE-01)
- Formato para empresas/una palabra: `EMPRESA-##` (ej: ESSALUD-01, CLÍNICAS-01)

**Lógica del Algoritmo**:
```typescript
// Si el nombre tiene más de una palabra (persona)
if (palabras.length > 1) {
  inicial = palabras[0].charAt(0)           // Primera letra del nombre
  palabraMasLarga = palabra más larga       // Usualmente el apellido
  prefijo = inicial + '.' + palabraMasLarga // C.AGUIRRE
}
// Si es una sola palabra (empresa o solo apellido)
else {
  prefijo = palabras[0]                     // ESSALUD
}

// Buscar siguiente número disponible
codigo = prefijo + '-' + numeroConCeros    // C.AGUIRRE-01
```

**Ejemplos Reales**:
- "Carlos Aguirre" → C.AGUIRRE-01, C.AGUIRRE-02, ...
- "Pedro Aguirre" → P.AGUIRRE-01, P.AGUIRRE-02, ...
- "Clínicas San Pablo" → CLÍNICAS-01 (palabra más larga)
- "Essalud" → ESSALUD-01
- "Alvarado" → ALVARADO-01

#### a) Auto-generación de Código
```typescript
const generarCodigoAutomatico = async (nombreCliente: string) => {
  const palabras = nombreCliente.trim().toUpperCase().split(' ')

  // Generar prefijo según cantidad de palabras
  let prefijo = ''
  if (palabras.length > 1) {
    const inicial = palabras[0].charAt(0)
    const palabraMasLarga = palabras.reduce((a, b) => a.length >= b.length ? a : b)
    prefijo = `${inicial}.${palabraMasLarga}`
  } else {
    prefijo = palabras[0]
  }

  // Buscar siguiente número disponible
  const { data: casosExistentes } = await supabase
    .from('casos')
    .select('codigo_estimado')
    .like('codigo_estimado', `${prefijo}-%`)
    .order('codigo_estimado', { ascending: false })
    .limit(1)

  let siguienteNumero = 1
  if (casosExistentes && casosExistentes.length > 0) {
    const ultimoCodigo = casosExistentes[0].codigo_estimado
    const match = ultimoCodigo.match(/-(\d+)$/)
    if (match) {
      siguienteNumero = parseInt(match[1]) + 1
    }
  }

  const nuevoCodigo = `${prefijo}-${String(siguienteNumero).padStart(2, '0')}`
  setFormData(prev => ({ ...prev, codigo_estimado: nuevoCodigo }))
}
```

- Genera código automáticamente al escribir el nombre del cliente
- Campo de código de solo lectura (no editable manualmente)
- Spinner de carga mientras genera el código


### Error 2: ON CONFLICT sin UNIQUE Constraint

**Error Original** (sesión anterior):
```
ERROR: 42P10: there is no unique or exclusion constraint matching the ON CONFLICT specification
```

**Causa**:
- Intentar usar `ON CONFLICT (codigo_estimado) DO NOTHING` en tabla `ubicaciones_fisicas`
- La tabla `ubicaciones_fisicas` NO tiene constraint UNIQUE en `codigo_estimado`
- Solo la tabla `casos` tiene UNIQUE constraint en ese campo

**Solución**:
```sql
-- ❌ INCORRECTO (para ubicaciones_fisicas)
INSERT INTO ubicaciones_fisicas (codigo_estimado, cliente, ...)
VALUES (...)
ON CONFLICT (codigo_estimado) DO NOTHING;

-- ✅ CORRECTO (sin ON CONFLICT)
INSERT INTO ubicaciones_fisicas (codigo_estimado, cliente, ...)
VALUES (...);

-- ✅ CORRECTO (solo para tabla casos)
INSERT INTO casos (codigo_estimado, cliente, ...)
VALUES (...)
ON CONFLICT (codigo_estimado) DO NOTHING;
```

**Lección Aprendida**:
- Solo usar `ON CONFLICT` en columnas con constraint UNIQUE o PRIMARY KEY
- Verificar constraints de cada tabla antes de usar ON CONFLICT
- La tabla `casos` tiene: `UNIQUE (codigo_estimado)`
- La tabla `ubicaciones_fisicas` NO tiene UNIQUE en `codigo_estimado`

### Error 3: Códigos Duplicados en Migración

**Error Encontrado** (esta sesión):
```
ERROR: 23505: duplicate key value violates unique constraint "casos_codigo_estimado_key"
DETAIL: Key (codigo_estimado)=(BARRAZA-01) already exists.
```

**Causa**:
- El script de migración intentaba actualizar múltiples registros al mismo código
- No se validaba si el código nuevo ya existía antes de intentar actualizar
- Múltiples casos generaban el mismo prefijo

**Solución Implementada**:
```sql
-- Detectar conflictos ANTES de actualizar
codigos_con_conflictos AS (
  SELECT
    cg.id,
    cg.codigo_viejo,
    cg.codigo_nuevo_propuesto,
    CASE
      WHEN EXISTS (
        SELECT 1 FROM casos c
        WHERE c.codigo_estimado = cg.codigo_nuevo_propuesto
          AND c.id != cg.id
      ) THEN true
      ELSE false
    END as tiene_conflicto
  FROM codigos_generados cg
)
-- Solo actualizar casos sin conflicto
UPDATE casos c
SET codigo_estimado = m.codigo_nuevo
FROM mapeo_codigos m
WHERE c.id = m.id
  AND c.codigo_estimado != m.codigo_nuevo
  AND m.tiene_conflicto = false;
```

**Mejoras Agregadas**:
1. Usar `id` como criterio de desempate en caso de misma fecha de creación
2. Validar conflictos ANTES de intentar UPDATE
3. Mantener código viejo si hay conflicto
4. Reportar qué casos no se pudieron migrar y por qué

**Lección Aprendada**:
- SIEMPRE validar unicidad antes de actualizar columnas con constraint UNIQUE
- Usar `id` como desempate para casos con misma `created_at`
- Implementar verificación de conflictos en migraciones complejas
- Mantener datos originales en caso de conflicto (no forzar cambios)
